## 1. Executive summary

The codebase is generally clean, well-commented, and the offline-first PWA architecture is coherent. The findings cluster into a few **systemic** issues rather than scattered one-offs, plus one genuinely urgent area: **backend authorization/security**.

Headline counts: **~145 findings** вҖ” roughly **41 рҹ”ҙ**, **40 рҹҹ **, **48 рҹҹЎ**, **16 рҹ”ө**.

The three things that most deserve attention:
- **Backend security cluster** (12 рҹ”ҙ in `ultistats_server/`): the entire authorization model hinges on one env var (`ULTISTATS_AUTH_REQUIRED`) whose default disagrees across config vs. guards; on top of that there is unauthenticated SSRF, path traversal from unvalidated IDs, several read endpoints with no authz, and non-atomic unlocked file writes that can lose or corrupt data under concurrent multi-coach use. **Confirm the production env first** вҖ” it determines how exploitable the rest is.
- **Stats correctness & entry-surface divergence** (frontend): player stats are keyed by *name* not *ID* in multiple places (silent stat-merge on duplicate names, split history on rename), and there are **five different event-entry surfaces** (Simple/Full/Field/Key Play/score-attribution) whose logic has already drifted вҖ” Key Play credits an assist on every completion, skips `completedPasses`, and never publishes to the narration bus, so the Full/Field tabs show stale state and stats genuinely differ by which screen the coach used.
- **Structural maintainability**: four god-files (`gameScreen.js` 5568, `teamSelection.js` 2325, `main.py` 2030, `main.css` 4412), an all-globals/manual-load-order wiring model (48 `<script>` tags, 235 `window.*` assignments, fixed `setTimeout(100)` init, monkey-patched globals), and heavy hand-duplication of deserialize/merge/render logic that has already produced real divergence bugs.

## 2. Systemic themes

These cut across multiple files; fixing the pattern is higher-leverage than fixing instances.

1. **рҹ”ҙ Stats keyed by player name, not ID.** `utils/eventStats.js` (accumulate by `playerName`), `teams/rosterManagement.js`, `teams/eventRoster.js` all key stats on `player.name`. Two players sharing a name merge; a rename splits history. The app already has stable IDs everywhere else вҖ” this is the single most impactful correctness theme on the frontend. (Migrating keys has data-migration implications вҖ” see Open Questions.)

2. **рҹ”ҙ Five divergent event-entry surfaces.** `pbpPossession` (Simple/Full/Field) is the shared core, but `playByPlay/keyPlayDialog.js` and `playByPlay/scoreAttribution.js` bypass it вҖ” building raw `Throw`/`Defense`, calling `addEvent` directly, never publishing to `narrationEventBus`. Result: stats diverge by surface and Full/Field tabs (which only repaint on bus events) go stale. Either route every surface through the core or retire Key Play.

3. **рҹҹ  Heavy copy-paste of serialize/deserialize/merge/render logic вҖ” already drifting.** Point/possession deserialization is near-verbatim in 3+ places (`sync.js`, `storage.js`); the `pendingNextLine` merge is duplicated with a *real divergence* (`refreshGameStateFromCloud` drops the `useSeparateLines` merge); Field vs Full PBP duplicate `reconstructState`/`manualHolder` handling and have drifted on point-boundary clearing; three roster-table renderers duplicate column logic; the backend storage modules duplicate ID-generation and `_index.json` load/save across ~11 files. Extract shared helpers.

4. **рҹҹ  God-files.** `game/gameScreen.js` (5568), `teams/teamSelection.js` (2325), `ultistats_server/main.py` (2030), `main.css` (4412). Each has clear internal seams to split along (documented in the per-area sections). Size is actively hiding bugs (e.g. `updateGameEventsModalState` defined twice).

5. **рҹҹ  All-globals wiring with implicit load-order coupling.** 48 ordered `<script>` tags, 235 `window.* =` assignments, `setTimeout(initializeApp, 100)` as a readiness proxy, `window.showScreen`/`window.updateControllerUI`/`window.moveToNextPoint` monkey-patched at module-load, and pervasive `typeof X !== 'undefined'` global guards. Reordering or slow-loading a script silently degrades to offline/empty states. A native ES-module migration (no bundler required) would remove this whole class of fragility. (Big call вҖ” see Open Questions.)

6. **рҹ”ҙ Offline-first is broken on the Teams surface.** Auth guards across `teams/*` treat "not authenticated" as "blocked" rather than "offline/degraded." The known create-team bug (`teamSelection.js:910-1018`) is one instance: the sign-in guard returns *before* the offline `createTeamOffline()` fallback it's supposed to reach. A single `canActOffline()` helper distinguishing "no session" from "Supabase unreachable" should be applied consistently.

7. **рҹ”ҙ Sync data-loss & resiliency gaps.** Last-write-wins full-state game sync can clobber a concurrent coach's points; `updatedAt` is mutated on the live object even when the sync fails offline (suppressing later real server updates); the sync queue has no retry cap (a permanently-failing item retries forever every 5s); and offline detection string-matches only Chrome's "Failed to fetch" (breaks on Safari/Firefox).

8. **рҹ”ҙ Backend file-storage concurrency & security.** Non-atomic, unlocked read-modify-write of entity files and `_index.json` (lost writes / corrupt JSON / dropped memberships); path traversal from unvalidated IDs into `Path(...)`; unauthenticated SSRF in `/api/proxy-image`; missing authz on player/event reads; unbounded version-backup growth + blocking git subprocess in the async path; in-memory controller state that breaks under multi-worker.

9. **рҹҹ  Timestamp inconsistency, three ways.** Field casing (`game_start_timestamp` snake vs `gameEndTimestamp` camel vs `game_end_timestamp` snake) risks silent active/ended mis-detection; epoch-ms vs ISO-8601 mixed in the `pendingNextLine` merge; and `increment-version.py` stamps local time but appends a fake `Z`. Normalize through single helpers / canonical field names.

10. **рҹҹ  Layout-in-JS sticky columns, duplicated.** `teams/rosterManagement.js` (`makeRosterColumnsSticky`, ~95 lines) and `ui/activePlayersDisplay.js` (`makeColumnsSticky`, ~100 lines) both hand-set inline styles, hardcoded non-themeable colors, and box-shadow borders, fighting the CSS layer вҖ” exactly the trap ARCHITECTURE.md warns about. Move to CSS `position: sticky` classes.

11. **рҹҹ  Service worker hazards.** Caches *everything* including error and cross-origin responses (a cached 404 served offline as valid); auto-reloads on SW activation with no guard (data loss / reload-loop mid-game); detects the API by substring host/port match (breaks `dev-backend.sh`'s `:8001+` ports); cache version bumped by hand each deploy.

12. **рҹҹЎ Dead/debug code shipped to production.** Deprecated shims (`importCloudGame`, `pullDataFromCloud`, legacy `selectTeam(index)`, etc.), stub `phase 5/7` handlers logging to console, the entire `FAST_PASS_EVENTS_ENABLED=false` narration path, a debug player-ID `<code>` block in the edit-player dialog, and 350 `console.*` calls across shipped frontend JS with no stripping.

### 1. Bugs / Correctness & Security

- рҹ”ҙ `config.py:68` / `auth/dependencies.py:176,237` / `jwt_validation.py:128` вҖ” **AUTH_REQUIRED defaults to false вҶ’ wide-open API**. The default is `"false"`, and the per-request guards re-read `ULTISTATS_AUTH_REQUIRED` defaulting to `"true"`, so behavior depends entirely on which env var is set at deploy. If the EC2 unit doesn't export `ULTISTATS_AUTH_REQUIRED=true`, every game/team/player/membership write endpoint is authorizable by *anyone* вҖ” and worse, in test mode any caller can become an arbitrary user just by sending `X-Test-User-Id`. The two layers also disagree on the default, which is a latent foot-gun. Fix: make the default `true` everywhere and read it from one shared config constant, not three independent `os.getenv` calls.

- рҹ”ҙ `main.py:269-293, 327-345, 373-397, 2000-2025` and `game_storage.py:243` etc. вҖ” **Path traversal via unvalidated IDs/filenames in file paths**. User-supplied `filename`, `game_id`, `timestamp`, `player_id`, `team_id`, `event_id`, `share hash`, and invite `code` are concatenated straight into `Path(...)` with no check that the resolved path stays under the intended directory. e.g. `GET /api/games/{game_id}` вҶ’ `GAMES_DIR / game_id / "current.json"`; a `game_id` like `../../../etc` (URL-encoded) or a `timestamp` like `../../current` can escape the games dir, and the static handlers do `pwa_dir / filename` after only checking the first path segment against a whitelist (so `game/../../secret` passes the `first_part in pwa_static_dirs` check). Fix: validate every ID against a strict `^[A-Za-z0-9_-]+$` pattern (and reject `..`), and for file serving resolve the path and assert `resolved.is_relative_to(base_dir)`.

- рҹ”ҙ `main.py:419-525` вҖ” **`/api/proxy-image` is an unauthenticated SSRF vector**. No auth dependency; it fetches any attacker-supplied `http(s)` URL server-side with `follow_redirects=True` and only checks the scheme. From EC2 this can hit `http://169.254.169.254/...` (instance metadata / IAM creds), internal services, or `localhost:8000` itself. The content-type/size checks happen *after* the request is made, so blind SSRF still works (timing, error differences). Fix: require auth, block private/link-local/loopback IP ranges after DNS resolution, disable redirects (or re-validate each hop), and consider an allowlist.

- рҹ”ҙ `main.py:686-714, 717, 741, 762` (events) вҖ” **Event read/list and create-team-membership authorization gaps tied to AUTH_REQUIRED**. `GET /api/events/{event_id}` and `GET /api/teams/{team_id}/events` require only *authentication*, not team access вҖ” any logged-in user can read any team's events/schedule. The create/update/delete event checks are gated behind `if AUTH_REQUIRED:` (line 697,730,753), so when auth is not required they silently skip the coach check rather than using the shared `require_*` dependencies the rest of the file uses. Fix: route these through `require_team_access` / `require_team_coach` dependencies (which centralize the admin-bypass + auth-disabled logic) instead of ad-hoc inline `if AUTH_REQUIRED` blocks.

- рҹ”ҙ `main.py:1679-1692, 1727-1760, 1809-1830 (get), 919-...` вҖ” **Several read endpoints have no authorization at all**. `GET /api/players`, `GET /api/players/{id}`, `GET /api/players/{id}/games`, `GET /api/players/{id}/teams`, and `GET /api/players/{player_id}` expose all player data (names, possibly contact info) to anyone, authenticated or not вҖ” no `Depends` at all. `POST /api/players` (line 1651) lets any authenticated user create/overwrite a player by supplying an existing `id` (line 1671 calls `update_player` with no ownership check). Fix: add `require_*` dependencies; gate player reads behind team access and player writes behind `require_player_edit_access`.

- рҹ”ҙ `game_storage.py:185-222` вҖ” **Blocking `subprocess.run` git calls inside async sync handler**. When `ENABLE_GIT_VERSIONING` is on, `save_game_version` (called from the async `sync_game` handler) shells out to git synchronously under `_SAVE_LOCK`, blocking the event loop for the duration of `git add`/`git commit` on every sync. Combined with the global lock this serializes all syncs across all games and stalls the whole server. Fix: run storage writes in a threadpool (`await run_in_threadpool(...)`) and/or drop git versioning in favor of the timestamped JSON files already written.

- рҹ”ҙ `game_storage.py:170-227` вҖ” **Version backups grow without bound**. Every sync writes a new `versions/{timestamp}.json` (full game state) and there is no pruning anywhere in the codebase. A live game pinged/synced every few seconds for hours produces thousands of full-state copies; over a season this fills the EC2 disk. Fix: cap retained versions (keep last N + daily snapshots) or prune on a schedule.

- рҹ”ҙ `game_storage.py:174` / `_write_game_version` вҖ” **Second-granularity timestamp collides under rapid sync**. Version filenames use `"%Y-%m-%dT%H-%M-%S"`; two syncs in the same second (very plausible with two coaches) overwrite the same version file and `current.json` is copied from it вҖ” silent loss of one version, and `restore` to that timestamp is ambiguous. Fix: add sub-second/counter suffix or use a monotonic sequence.

- рҹ”ҙ `auth/dependencies.py:185-195` вҖ” **`require_game_team_coach` consumes the request body, which can break the handler**. It calls `await request.json()` to read `teamId` for new games; FastAPI usually re-buffers, but relying on reading the stream in a dependency and again in the endpoint (`game_data: Dict = Body(...)`) is fragile and order-dependent. Also, for a *new* game the caller supplies `teamId` in the body, so a user who is a coach of team A can create a game claiming `teamId` of team A while the game content references team B вҖ” the teamId is trusted verbatim. Fix: read the parsed body via a shared model/dependency, and validate teamId consistency.

- рҹ”ҙ `controller_storage.py` (whole module) вҖ” **In-memory controller state is process-local; multi-worker deployments break it**. All role/handoff/connected-coach state lives in module globals guarded by a `threading.Lock`. If the server ever runs more than one uvicorn worker (or behind multiple processes), two coaches hitting different workers see divergent "Active Coach" state, and the `authoritative` ownership check in `sync_game` (main.py:815) becomes unreliable вҶ’ play data can be rolled back. Fix: document/enforce single-worker, or move controller state to a shared store (Redis).

- рҹ”ҙ `main.py:560-580` `PATCH /api/auth/me` вҖ” **`update_user_storage` does a read-modify-write with no lock**; concurrent profile updates (or any user-file writer like `create_or_update_user` on every `/auth/me`) can interleave and lose writes. Same non-atomic pattern in every storage module (write to the live file directly, no temp-file+rename). A crash mid-`json.dump` leaves a truncated/corrupt JSON file that subsequent reads will fail on. Fix: write to a temp file and `os.replace()` atomically; add per-entity locking for read-modify-write.

- рҹ”ҙ `membership_storage.py:59-109` / `invite_storage.py` / `share_storage.py` вҖ” **`_index.json` read-modify-write has no lock and isn't atomic**. `_update_index_add/remove` load the whole index, mutate, and rewrite. Two concurrent membership creations (e.g. two users redeeming invites at once) race: both read the old index, each writes back its own version, and one membership is dropped from the index вҶ’ that user silently loses team access until a manual rebuild. Same risk for invite/share indexes. Fix: serialize index writes with a lock and write atomically.

### Sync layer вҖ” `store/sync.js`

- рҹ”ҙ `store/sync.js:204-254` вҖ” **Sync queue mutated during iteration; failed item gets re-queued by later success.** `processSyncQueue` iterates over a `sortedQueue` snapshot but mutates the live `syncQueue` via `filter` inside the loop, and on a non-network error simply increments `retryCount` and keeps going. There is no retry cap, so a permanently-failing item (e.g. server 400/422 from a malformed payload) is retried forever every 5s, and `retryCount`/`lastError` accumulate unbounded. Add a max-retry threshold that drops or quarantines the item (and surfaces it to the user) instead of looping indefinitely.

- рҹ”ҙ `store/sync.js:241-244` вҖ” **Network-error detection is brittle and only catches one browser's wording.** Mid-queue network failures are detected by `error.name === 'TypeError' && error.message === 'Failed to fetch'`. Safari/Firefox use different messages ("Load failed", "NetworkError when attempting to fetch resource"), so on those browsers a dropped connection is treated as a hard server error: the item's `retryCount` is bumped and the loop keeps hammering the next items instead of stopping. Detect offline via `!navigator.onLine` or a broader check rather than string-matching one Chrome message.

- рҹ”ҙ `store/sync.js:163-179,309-318` вҖ” **Last-write-wins game sync coalesces queued game writes вҖ” offline edits can clobber a concurrent coach's points.** `addToSyncQueue` removes any existing pending item for the same `{type,id}` and the game action is always `'sync'` with the full game state. Two devices editing the same game offline, then coming online, each POST their entire game to `/sync`; whichever lands last wins and the other's points/events are lost. There's no per-field merge on the game body itself (only `pendingNextLine` is field-merged, and only on *read*). Document this as accepted (AC is authoritative) or add server-side point-level merge; at minimum flag it for the owner.

- рҹ”ҙ `store/sync.js:455-456,596-597` вҖ” **`syncPlayerToCloud`/`syncTeamToCloud` mutate the caller's live object's `updatedAt` even when offline.** They set `player.updatedAt = new Date().toISOString()` on the passed-in object before queueing. If the sync later fails or is offline, the in-memory/localStorage object now has a future `updatedAt` that will beat a genuinely-newer server record in the `serverUpdated > localUpdated` comparison in `syncUserTeams` (line 1438), suppressing a real incoming update. Bump `updatedAt` only on the serialized copy, not the source object.

- рҹ”ҙ `store/sync.js:1112-1133,1266-1284` вҖ” **`pendingNextLine` merge mixes epoch-ms and ISO-8601 timestamps inconsistently.** Line-type fields use `new Date(modKey).getTime()`, but `lineupReadyAt`/`localReadyAt` are compared as raw values (`serverReadyAt > localReadyAt`) assuming epoch ms (models.js documents `lineupReadyAt` as "epoch ms"). `lineCoachViewingAt`/`useSeparateLinesAt` are parsed via `new Date(...).getTime()` (ISO). If any writer ever stores `lineupReadyAt` as an ISO string (e.g. via `Date.now()` vs `toISOString()` drift across the codebase), the bare `>` comparison silently mis-orders. Normalize all timestamp comparisons through one `toMs()` helper.

- рҹҹ  `store/sync.js:1186-1321` / `1083-1177` вҖ” **`refreshGameStateFromCloud` and `refreshPendingLineFromCloud` duplicate ~60 lines of point/possession/event deserialization and pendingNextLine merge logic.** The point-rebuild block (1219-1250) is a near-verbatim copy of the one in `loadGameFromCloud` (1047-1066) and `storage.js deserializeGame` (450-464); the pendingNextLine merge (1112-1156 vs 1266-1296) is duplicated with subtle differences (the second copy drops the `useSeparateLines` merge entirely вҖ” a real divergence bug). ### Bootstrap & service worker (main.js, navigation.js, service-worker.js)

- рҹ”ҙ `main.js:374` вҖ” **App init fires on a fixed 100ms `setTimeout`, not on module-load readiness.** `setTimeout(initializeApp, 100)` assumes every dependency (`window.breakside.auth`, `showSelectTeamScreen`, etc.) is loaded within 100ms. On slow devices or a cold cache this races: if `auth.js` hasn't defined `window.breakside.auth.initializeAuth` yet, `initializeApp` silently falls into the "Auth module not loaded, running in offline mode" branch (line 243) and a logged-in user is dropped to offline mode. Replace the magic timer with an explicit readiness signal (e.g. run `initializeApp` from the same `DOMContentLoaded`/`load` flow that guarantees script order, or poll for the symbol).

- рҹ”ҙ `main.js:93-97` & `main.js:88-99` вҖ” **Service-worker `updatefound`/`statechange` auto-reloads the page with no guard, risking a reload loop and data loss mid-game.** When a new SW activates, `window.location.reload()` is called unconditionally. A coach mid-point would lose unsaved in-memory state and the live narration WebSocket. There's no "update ready, tap to refresh" prompt and no check that it's safe to reload. Gate the reload behind a user prompt (or skip while a game/recording is active) вҖ” `controllerchange` is also the more correct event than `installing` statechange here.

- рҹҹ  `service-worker.js:42-67` вҖ” **Network-first `fetch` handler caches *everything*, including error responses and opaque/cross-origin responses, with no status check.** `caches.put` runs on any `networkResponse` whose request was GET and not API вҖ” a 404 or 500 HTML page gets cached and then served offline as if valid. It also caches third-party CDN assets (Supabase JS, FontAwesome) opaquely. Only cache `networkResponse.ok` same-origin responses. Note also the cache name `build-254` (line 1) must be bumped by hand each deploy or stale assets persist (the CLAUDE.md cherry-pick caveat confirms this is an active footgun).

- рҹҹ  `service-worker.js:35-39` вҖ” **API bypass is substring matching on hardcoded host fragments.** `url.includes(':8000/')`, `'api.breakside.pro'`, `'api.breakside.us'`. A dev backend on a different port (the `dev-backend.sh` flow picks a free port from 8000+, e.g. `:8001`) would be cached as a static asset, breaking sync against the local server. Match the API by a robust predicate (path prefix `/api/` or an allowlist of API origins) rather than ad-hoc port strings.

- рҹҹ  `main.js:159` вҖ” **`window.location.reload(true)` uses the long-deprecated forced-reload boolean.** `reload(true)` is ignored by modern browsers (no-op argument); the cache clear above it is what actually matters. Drop the argument to avoid implying behavior that no longer exists.

- рҹҹЎ `main.js:570` вҖ” **Unconditional `document.getElementById('countdownTimer').style.display` will throw if the element is absent.** Unlike the surrounding code that null-checks every lookup, this line assumes `countdownTimer` exists; a missing element throws and aborts the rest of `DOMContentLoaded` (including `matchButtonWidths`). Add the same `?.`/null guard used elsewhere.

- рҹҹЎ `navigation.js:64-70` вҖ” **`manageControllerButtons` ignores its `screenId` param and always hides.** The function takes `screenId`, documents per-screen logic, then unconditionally calls `setControllerButtonsVisible(false)`. Dead parameter / misleading abstraction вҖ” inline the single call or drop the param.

### Auth (auth/auth.js, auth/loginScreen.js)

- рҹ”ҙ `auth.js:213-214` & comment вҖ” **`getAuthHeaders` claims `getSession()` "handles token refresh automatically" вҖ” it does not.** The Supabase v2 `getSession()` returns the cached session and only refreshes if it has already expired *and* autoRefresh ran; it does not proactively refresh. For a long sideline session where the access token expires while offline-then-online, API calls can go out with a stale/expired bearer and 401. Use `supabaseClient.auth.getUser()` or explicitly check `expires_at` and call `refreshSession()` when near expiry. At minimum, retry once on 401 by forcing a refresh.

- рҹҹ  `auth.js:88-109` вҖ” **`syncUserTeams` on session-restore is fire-and-forget inside a `setTimeout(500)` with no coordination with `initializeApp`.** main.js calls `hideAuthScreenAndShowApp()` immediately after `initializeAuth()` resolves, but team sync happens 500ms later asynchronously; the team-selection screen can render empty then suddenly repopulate (the code even re-calls `showSelectTeamScreen()` to paper over this). The 500ms delay is a magic number masking an ordering problem. Await team sync as part of init, or drive the UI off a sync-complete event.

- рҹҹ  `auth.js:116` вҖ” **`onAuthStateChange` overwrites `currentSession`/`currentUser` even for spurious `null` sessions.** Supabase fires `INITIAL_SESSION`/`TOKEN_REFRESHED` etc.; on certain transient events `session` can be null and this clobbers a valid in-memory session, flipping `isAuthenticated()` to false mid-use. Guard against overwriting a live session with null except on explicit `SIGNED_OUT`.

- рҹҹ  `auth.js:159` вҖ” **Test-mode session injection is reachable in production via `?testMode=true`.** `enableTestMode` sets a fake authenticated session and `getAuthHeaders` sends `X-Test-User-Id`. main.js (line 179) gates this only on the URL param, with no build/env check. If the backend honors `X-Test-User-Id` in prod (controlled by `ULTISTATS_AUTH_REQUIRED`), this is an auth-bypass. Confirm the header is rejected in production and/or compile test-mode out of prod builds. рҹ”ө вҖ” needs owner confirmation of the server-side guard.

- рҹҹЎ `auth.js:401-451` вҖ” **Two parallel global export objects (`window.BreaksideAuth` and `window.breakside.auth`) expose overlapping APIs.** The "legacy" object lacks the sign-in functions; callers can pick the wrong one. Consolidate to one namespace and alias the other for one release, then remove.

- рҹҹЎ `auth.js:235-247` вҖ” **`getAccessToken` duplicates `getAuthHeaders` session logic.** Both call `getSession()` and unwrap the token. Factor a single `getFreshSession()` helper so a future refresh fix lands in one place.

- рҹҹЎ `loginScreen.js:127-140` вҖ” **`setAuthLoading` stomps `dataset.originalText` if called twice while loading.** A second `setAuthLoading(true)` (e.g. double-submit) records "Loading..." as the original text, so the button stays "Loading..." after completion. Guard with `if (loading && !btn.dataset.originalText)`.

### Landing & invite-join (landing/landing.js, landing/join.js)

## 6. Conflict-aware execution plan

The findings in Section 5 are re-sliced here into **tasks designed to minimize merge conflicts** between parallel sessions. The guiding principles:

- **Separate bug fixes from refactors.** Small surgical fixes rarely collide even in the same file; the *structural* changes (splitting god-files, deduping renderers, ES modules) are the conflict generators. Land bug fixes first, refactor against the updated `main`.
- **One task owns a directory/file.** Two sessions editing the same god-file вҖ” especially if one is *splitting* it вҖ” is unmergeable. Each file has exactly one owner per phase.
- **ES modules is solo and last.** It rewrites `import`/`export` across all ~48 files; any concurrent frontend branch becomes a brutal rebase.
- Use the CLAUDE.md worktree workflow (`git worktree add .worktrees/<task> -b <task>`); rebase later branches onto `main` after earlier ones merge.

### Run order at a glance
